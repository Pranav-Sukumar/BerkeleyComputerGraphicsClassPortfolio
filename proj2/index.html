Project Overview

In this project, we learned about the core aspects of geometric modeling and worked on implementing functionality to compute and render various objects.

In Section 1, our main goal was to implement Bezier curves and surfaces. For task 1, we used various anchor points in order to evaluate a 2D Bezier curve using de Casteljau’s algorithm. For task 2, we built off of this concept and took it a step further by evaluating 3D Bezier surfaces by using the separable 1D de Casteljau algorithm.

In Section 2, we moved onto the bigger picture of triangle meshes and worked with the core data structures of half-edges, edges, vertices, and faces. For task 3, we implemented area-weighted vertex normals which are used to improve and even out the shading across our geometric model without appearing discrete and tessellated. For task 4 and task 5, we implemented the core edge flip and split functions respectively, which are useful for modifying the individual triangles within our geometric model. Finally, for task 6, we used the previous tasks to implement loop subdivision for mesh upsampling, which essentially allows us to create and subdivide more fine-grained triangles within our existing geometric model, in order to make it appear much more smooth, detailed, and seamless as the number of triangles increases while their size decreases.

Part 2: Bezier Surfaces with Separable 1D de Casteljau

How the de Casteljau algorithm extends to Bezier surfaces, and how we implemented it in order to evaluate Bezier surfaces

We can extrapolate off our previous approach in order to implement the de Casteljau algorithm for Bezier surfaces as well through two variables uuu﻿ and vvv﻿. We first iterate over all 3D points to evaluate Bezier curves in uuu﻿ (through our evaluate1D() function which repeatedly calls evaluateStep()), and retrieve a single control point for each curve in uuu﻿. Then, we connect these new points by evaluating the Bezier curve in vvv﻿, to finally get our ultimate point in (u,v)(u, v)(u,v)﻿ that lies on the Bezier surface (through our evaluate() function).

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-Weighted Vertex Normals

How we implemented the area-weighted vertex normals

To implement the area-weighted vertex normals in Part 3, we first decided to define two variables: halfedge_base to store the initial first halfedge and halfedge_traverse to reassign new halfedges as we explore neighboring triangles for the vertex. For each of these triangles, we used a correct mix of next(), vertex(), and twin() functions in order to retrieve all three vertices for each triangle we explored. Then, we computed the area of each triangle by taking the cross product of two vectors among the three vertices, and dividing the norm of this cross product vector by 2. Thereafter, we multiply this area by the face normal of the triangle, and we maintain a running sum of all these Vector3D values for every triangle that shares our original vertex. Finally, we divide the resulting summed Vector3D by its norm to get our output for the area-weighted vertex normal.

Part 6: Loop Subdivision for Mesh Upsampling

How we implemented loop subdivision

We implemented the loop subdivision algorithm by following the pseudocode outlined in the starter code and by following the formulas mentioned on the spec.

Loop subdivision allows us to upsample a 3D mesh, similar to how bilinear filtering was used to upsample 2D images in Project 1. Loop subdivision performs 4-1 subdivisions over the entire mesh by splitting all edges, and flipping them back. The algorithm also requires that both the new and old vertices are repositioned accordingly.

The algorithm can be explained in depth by looking at the code we wrote that was split up into 5 stages:

  1. We first looped over all vertices in the mesh to precompute the eventual new position for all existing vertices. For each vertex that we looped over, we first computed the original_neighbor_position_sum by looping over all neighboring vertices. We then set the current vertex’s precomputed newPosition using the formula in the spec (and in the code block below). We also set the vertex’s isNew attribute to false.
currVertex->newPosition = (1 - n * u) * currVertex->position 
+ u * original_neighbor_position_sum;
  1. We then looped over all edges in the mesh to precompute the eventual new position for all new vertices resulting from an eventual split (note we did not perform the actual split in this step). For each edge that we looped over, we first determined the neighboring A,B,C,D vertices. We then set the current edge’s precomputed newPosition using the formula in the spec (and in the code block below). We also set the edge’s isNew attribute to false.
currEdge->newPosition = 3.0/8.0 * (A->position + B->position) 
+ 1.0/8.0 * (C->position + D->position);
  1. We then looped over all edges in the mesh to split each one. We ensured we were only splitting edges that were new to prevent infinite looping. The new edges are labeled as new from our code in Task 5.
  1. We then looped over all the new edges in the mesh to flip any new edge that connected a new vertex to an old one.
  1. We finally looped over all vertices in the mesh to assign each vertex its correct position which was precomputed in steps 1 and 2.

When we subdivide the base cube, it becomes asymmetrically shaped over time because of the fact that the position of a new vertex resulting from a split is determined by the edge’s 4 neighbor vertices. Since the cube with no preprocessing only has one diagonal edge per face, this results in an asymmetric location for the new vertex resulting from the splitting.

Asymmetric effects when subdividing the base cube:

To fix this we can preprocess the cube by splitting the diagonal edge on each of the 6 faces. Now each face has a vertex that is equidistant from the 4 vertices on the face. This means that the surrounding vertices are more equidistant. When an edge is split, the position of the new vertex is determined by the position of its 4 neighbors, and now that these 4 neighbors are more equidistant, the vertex’s new position is more symmetric.
